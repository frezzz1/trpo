ADR: 2025-12.3 - Выбор технологического процесса разработки

Дата: 10 декабря 2025 г.
Статус: Принято

Контекст:

Для успешной разработки и поддержки системы управления лояльностью клиентов необходимо определить технологический процесс, который обеспечит:

Эффективное управление требованиями и приоритетами

Быстрое внедрение новых функций при сохранении качества

Прозрачность процесса для всех участников

Возможность адаптации к изменяющимся требованиям бизнеса

Контроль качества и безопасности на всех этапах

Поддержку модульной архитектуры и командной работы

Рассмотренные варианты:

Гибкая методология (Agile/Scrum) — итеративная разработка с короткими циклами, регулярными демо и обратной связью

Kanban — непрерывный поток работы с визуализацией и ограничениями WIP

Гибридный подход (Scrumban) — сочетание структурированности Scrum и гибкости Kanban

Waterfall с элементами Agile — последовательная разработка с этапами, но с возможностью адаптации

Решение:

Выбран гибридный подход Scrumban с элементами Agile и DevOps practices.

Обоснование:

Структурированность для сложного проекта:

Еженедельные спринты для планирования и синхронизации команды

Регулярные демо и ретроспективы для непрерывного улучшения

Четкие роли и ответственности (Product Owner, Scrum Master, разработчики)

Гибкость для динамичных требований:

Непрерывный поток работы через Kanban-доску

Возможность внесения критических изменений вне спринтов

Приоритизация по бизнес-ценности и рискам

DevOps интеграция:

CI/CD pipeline как часть процесса разработки

Автоматизированное тестирование и развертывание

Мониторинг и обратная связь от production

Ключевые практики процесса:

Управление требованиями:

User Stories с четкими критериями приемки

Эпики для крупных функциональных блоков

Технические задачи для долга и рефакторинга

Планирование и оценка:

Еженедельное планирование спринтов

Оценка сложности через story points

Capacity planning с учетом отпусков и обучения

Разработка:

Feature branches с code review

Парное программирование для сложных задач

Автоматизированное тестирование

Статический анализ кода

Контроль качества:

Definition of Done для каждой задачи

Автоматические проверки в CI/CD

Ручное тестирование критических сценариев

Регулярные security reviews

Внедрение:

Автоматическое развертывание в staging

Canary releases для новых функций

Feature flags для управления функциональностью

Blue-green deployment для минимизации downtime

Рабочие артефакты:

Product Backlog — приоритизированный список всех требований

Sprint Backlog — задачи на текущий спринт

Kanban Board — визуализация текущего статуса работ

Definition of Ready — критерии готовности задачи к разработке

Definition of Done — критерии завершенности задачи

Роли и ответственности:

Product Owner — управление backlog, приоритизация, коммуникация с бизнесом

Scrum Master — фасилитация процесса, удаление блокеров

Development Team — кросс-функциональная команда разработки

QA Engineer — обеспечение качества, тестирование

DevOps Engineer — инфраструктура, CI/CD, мониторинг

Преимущества выбранного подхода:

Баланс между структурированностью и гибкостью

Прозрачность процесса для всех заинтересованных сторон

Быстрая обратная связь и адаптация

Поддержка непрерывного улучшения

Масштабируемость для растущей команды

Недостатки и риски:

Сложность внедрения для команды без Agile-опыта

Риск превращения в "Agile-театр" без реальных улучшений

Требует дисциплины и вовлеченности всех участников

Смягчающие меры:

Поэтапное внедрение с обучением команды

Регулярные ретроспективы для улучшения процесса

Адаптация процесса под специфику команды и проекта

Использование инструментов для автоматизации рутинных задач

Инструментальная поддержка:

Jira — управление задачами и backlog

Confluence — документация и knowledge base

GitLab CI/CD — автоматизация сборки и развертывания

Slack/Teams — коммуникация и интеграции

SonarQube — контроль качества кода

Критические этапы процесса:

Инициация: Согласование требований, оценка, создание эпиков

Планирование: Приоритизация, разбивка на задачи, планирование спринта

Разработка: Кодирование, тестирование, code review

Внедрение: Сборка, развертывание, мониторинг

Поддержка: Обработка инцидентов, оптимизация, рефакторинг

Последствия:

Для команды:

Регулярное планирование и синхронизация

Прозрачность задач и прогресса

Непрерывное обучение и улучшение навыков

Для качества:

Встроенные проверки на всех этапах

Раннее обнаружение и исправление дефектов

Систематический подход к тестированию

Для бизнеса:

Предсказуемость сроков и результатов

Возможность быстрой адаптации к изменениям

Регулярная демонстрация прогресса

Для продукта:

Постепенное наращивание функциональности

Высокая надежность и безопасность

Легкость поддержки и развития

Метрики успеха:

Velocity команды стабилен или растет

Cycle time задач ≤ 5 дней

Время восстановления после инцидентов ≤ 1 часа

Customer satisfaction ≥ 90%

Технический долг под контролем и уменьшается
